# Frontend Architecture

The SPA is React 19 + TypeScript + Vite — served by Vite in dev (proxying `/api`, `/openapi`, `/scalar` to the backend on `:5001`) and from the .NET app's `wwwroot` in production. Server state is **TanStack Query**, client state is **Zustand**, routing is **TanStack Router** (file-based), UI is **MUI** (single dark theme). The fetch client and every API hook are generated from the OpenAPI spec — see `API_Integration.md`.

Pinned versions live in `ClientApp/package.json` — don't transcribe them here, they drift. Notable libraries: `@mui/material` + `@mui/x-date-pickers`, `@tanstack/react-router` + `@tanstack/react-query`, `firebase`, `@dnd-kit/*` (drag-and-drop), `i18next` (+ `i18next-http-backend`), `@fullcalendar/*` (expiry calendar), `sonner` (toasts), `vite-plugin-pwa`, `@grafana/faro-*` (RUM). The Vite build runs the **React Compiler** (`babel-plugin-react-compiler`; see `Performance_Optimization.md`).

## Project structure

Vertical-slice `features/<area>/` folders mirror the backend slice folders one-to-one. **New work belongs in `features/`.**

```
src/
├── features/                    # Vertical slices (canonical)
│   ├── households/              # one hook file per backend slice + pages/ + components/
│   │   ├── householdRole.ts     #   role tokens, color/label maps
│   │   ├── onboardingSkip.ts
│   │   ├── useUserHouseholds.ts  useCreateHousehold.ts  …
│   │   ├── pages/  components/
│   │   └── members/             # nested sub-slice (mirrors backend Households/Members/)
│   ├── lists/                   # + items/ sub-slice
│   ├── inventories/             # + items/, settings/, notifications/ sub-slices
│   ├── recipes/                 # + items/, sections/, links/, attachments/, copyToList/
│   ├── notifications/  settings/  me/activeHousehold/
├── routes/                      # File-based routing — thin shells → features/<area>/pages
│   ├── __root.tsx               #   root layout
│   ├── index.tsx                #   dashboard (+ first-run onboarding redirect)
│   └── auth/  household/  lists/  inventories/  recipes/  settings/
├── components/                  # cross-cutting primitives
│   ├── dialogs/ (ConfirmDialog)  layout/  shared/
│   ├── sortables/               #   generic dnd-kit wrappers (SortableList, SortableListItem)
│   └── inputs/                  #   shared item-input primitives (AddInput, QuantityPanel, DateInputPanel)
├── hooks/                       # cross-cutting hooks (useDebouncedInvalidation, useLongPress, …)
├── lib/
│   ├── openapi.json             #   emitted by the backend build
│   └── api/                     #   generated client — types.gen.ts, sdk.gen.ts, client.gen.ts, @tanstack/react-query.gen.ts
├── theme.ts                     # MUI theme + shared sx (see Frontend_Styling.md)

```

Two things deliberately **don't** exist: there are no `*Keys.ts` query-key factories (keys are generated), and there is no `_protected.tsx` layout route (auth is per-route). Both are covered below. `routeTree.gen.ts` is generated by `@tanstack/router-plugin/vite` — don't edit it by hand.

## Feature-folder rules

One backend slice → one frontend hook file; one backend feature folder → one `features/<area>/` folder; nested backend resources → nested sub-folders.

1. **One hook file per backend slice**, named for the verb (`useCreateList.ts`, `useListItems.ts`). No bundled `useXxxQueries.ts`.
2. **Never hand-roll query keys.** hey-api generates `getXOptions()` / `xMutation()` / `getXQueryKey()`; spread those. `*Keys.ts` factory files are forbidden — generated keys carry `tags` for invalidation (`API_Integration.md`).
3. **Pages in `pages/`, components in `components/`.** Route files are thin shells: `createFileRoute` + `requireAuth` + import the page. No fetching or layout logic in routes.
4. **Import types from `lib/api` directly** (`import type { ListItemResponse } from ".../lib/api/types.gen"`), not via a hook re-export — the codegen owns the names.
5. **Nest a sub-slice** only when the backend nests, the URL is parent-scoped, AND the sub-area has its own hook surface (e.g. `features/lists/items/`). Don't nest just to group files.
6. **Shared primitives live in `src/components/`** until they have one owner — `components/inputs/AddInput` is used by both lists and inventories.
7. **Bundle style cleanup with the move** — apply the `Frontend_Styling.md` checklist when pulling a component into `features/`.

Canonical references: `features/lists/useList.ts` (query), `features/lists/useDeleteList.ts` (mutation), `features/lists/items/useCreateListItem.ts` (optimistic), `routes/household/create.tsx` (route shell).

## Server state (TanStack Query)

Global defaults in `main.tsx` (`staleTime` ~5 min, `gcTime` ~10 min); each hook sets its own `staleTime`. Every hook spreads generated helpers — never write `queryFn` / `mutationFn` / a literal `queryKey`.

**Query** — spread `getXOptions({ path })`, guard `enabled` on each path id being `> 0`:

```ts
export const useList = (householdId: number, listId: number) =>
    useQuery({
        ...getListOptions({ path: { householdId, listId } }),
        enabled: householdId > 0 && listId > 0,
        staleTime: 1000 * 60 * 2,
    });
```

**Mutation** — arg-less; the caller passes `{ path, body }` to `mutate`/`mutateAsync`. Invalidation reads `variables.path.*` and rebuilds keys via `getXQueryKey({ path })`:

```ts
export const useDeleteList = () => {
    const queryClient = useQueryClient();
    return useMutation({
        ...deleteListMutation(),
        onSuccess: (_data, variables) => {
            queryClient.removeQueries({ queryKey: getListQueryKey({ path: variables.path }) });
            queryClient.invalidateQueries({
                queryKey: getListsQueryKey({ path: { householdId: variables.path.householdId } }),
            });
        },
    });
};
```

**Optimistic mutation** (hot paths — create/toggle/reorder items) keeps its `onMutate`/`onError`/`onSuccess` callbacks; those are the substance codegen doesn't replace. Patch the cache in `onMutate`, reconcile in `onSuccess`, coalesce refetches via `useDebouncedInvalidation`. Two rules from hard experience: (a) the optimistic patch must mirror the server's derived values; (b) a create hook **must swap its `Date.now()` temp id for the real server id in `onSuccess`**, or a later edit PUTs to a temp id that overflows `{itemId:int}` and 500s. Canonical: `features/lists/items/useCreateListItem.ts`.

`staleTime` cheatsheet: ~5 min slow-moving (households), ~2 min normal reads (lists/inventories), ~30 s hot collaborative reads (an open list's items).

## Routing & auth

File-based routing; `routeTree.gen.ts` is generated. Auth is **per-route, not a layout** — each protected route's `beforeLoad` calls `requireAuth` (`src/common/authGuard.ts`), which reads the Zustand auth store and redirects to `/auth/login` when there's no Firebase user. (There is no `_protected` layout route.) First-run onboarding: `routes/index.tsx` redirects a signed-in user with no households to `/onboarding` unless they've skipped (`features/households/onboardingSkip.ts`). Auth wiring: `Firebase_Auth_Setup.md`.

## Client state (Zustand)

Zustand holds local/session UI state only — the auth store (`common/authProvider.ts`), modal/selection state, view preferences. The **active household** is server-side session state reached through the API, not a client store. Don't introduce a third state layer (Redux, Context-as-store) for new features.

## Styling · performance · i18n · PWA

- **Styling** — `theme.ts` owns radius, responsive typography, and button overrides; the authoring rules + anti-pattern checklist are in `Frontend_Styling.md` (authoritative — trust it over inline sx in legacy routes).
- **Performance** — list-rendering optimizations (fractional-index ordering, checked-items windowing, drag handling, React Compiler) are in `Performance_Optimization.md`.
- **i18n** — `i18next` + `i18next-http-backend`; translations under `public/locales/{en,de}/translation.json`. A brand-new top-level namespace must be added to both locales AND registered in `src/types/i18next.d.ts`, or `tsc` rejects its keys. Tests never assert translated text.
- **Push · PWA · RUM** — push-only service worker (`src/sw.ts`) + `common/pushNotifications.ts` (`Push_Notifications.md`); install wiring in `common/pwa.ts`; Grafana Faro in `common/observability.ts` (`Observability.md`). Every `VITE_*` var must be declared as `ARG`+`ENV` in the Dockerfile `build_frontend` stage or it silently no-ops in the bundle.

## Testing

No frontend-only test runner (no Jest/Vitest/RTL). UI behaviour is covered end-to-end by Reqnroll/Playwright scenarios under `Frigorino.IntegrationTests/Slices/<Area>/<Feature>.feature`, driving real Chromium against the Vite build + a Postgres testcontainer. Assert on `data-testid` / `data-*`, never translated text. Static guardrails: `npm run tsc` + `npm run lint` + a successful `vite build`. See `Testing.md`.
