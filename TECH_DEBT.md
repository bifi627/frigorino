# Tech debt

Running list of known issues we've spotted but consciously deferred. Add new items as they're noticed; remove them once fixed (don't mark as done — git history covers that).

Format per item:

- **Title** — one-line hook.
- **Where:** file path(s) / line refs.
- **Why deferred:** the reason it didn't get fixed in the originating change.
- **Plan:** the fix sketch, concrete enough that a future contributor doesn't re-investigate from scratch.
- **Risk if left:** what could go wrong while it's still here.

## - **Automated database backup runner** — no database backups exist today; a single bad migration or accidental delete is unrecoverable.
- **Where:** Railway Postgres service (managed; `ConnectionStrings__Database` on the `Frigorino.Web` service). No backup automation in the repo or Railway config. Relevant in-process scaffolding for a self-hosted approach: `IMaintenanceTask` + `MaintenanceHostedService` (`Frigorino.Infrastructure/Services/`), which already run on cold start.
- **Why deferred:** the app is pre-/early-production and the data is reconstructable, so it hasn't bitten yet — but it's a latent single point of failure that wants a deliberate change, not folding into a feature.
- **Plan:** weigh (1) Railway's managed/native Postgres backup feature if/when available on the current plan (preferred — no app code, point-in-time or scheduled snapshots, offloaded retention); vs (2) a self-hosted dump runner — `pg_dump --no-owner --no-privileges` streamed to off-box object storage (e.g. the existing blob storage area or an S3-compatible bucket), with dated keys + a retention sweep. Because Railway's serverless tier sleeps on idle, a wall-clock scheduler would silently miss its window (see [[project_no_synthetic_uptime_checks]] / background-jobs note in CLAUDE.md), so a self-hosted runner should piggyback on cold-start `IMaintenanceTask` with a "skip if last backup < N hours" guard, OR run as an external cron outside the sleeping container (a separate Railway cron service / GitHub Actions scheduled job hitting the DB directly). Decide retention window + restore-test cadence (an untested backup isn't a backup).
- **Risk if left:** total/partial data loss with no recovery path — a bad migration, `ExecuteDelete` bug, accidental drop, or Railway-side incident wipes real client data ([[project_branch_workflow]]: stage hosts a real client) with nothing to restore from.

## - **Sync prod DB → stage for a realistic stage environment** — stage currently runs on its own thin/agent-seeded data, so it doesn't surface real-data issues before they hit prod.
- **Where:** Railway Postgres services for the stage and prod environments (`ConnectionStrings__Database` on each env's `Frigorino.Web` service). No copy/refresh mechanism exists. Ties into [[project_branch_workflow]] (stage→main FF promotion; stage hosts real client UAT) and the per-feature/per-env blob areas (`refactor(storage): per-feature/per-env blob areas`).
- **Why deferred:** stage works for UAT today on seeded data; building a safe one-way refresh is its own change, and copying real prod data into a lower environment raises privacy/secrets handling that wants deliberate thought, not folding into a feature.
- **Plan:** one-directional **prod → stage only** refresh (never the reverse — see risk). Likely a one-shot `pg_dump`/restore (same shape as the PG16→18 side-by-side runner documented below) repointed at stage's `DATABASE_URL`, run on demand rather than scheduled. Decide: (1) whether to **anonymize/scrub** PII before loading into stage (real client data in a lower env — emails, household names) or accept that stage mirrors prod verbatim; (2) handle **blob storage** alongside the DB — per-feature/per-env areas mean DB rows reference blobs in the *prod* area, so stage rows would point at prod blobs unless those are copied/repointed too; (3) preserve or re-seed the `dev-user`/stage test identities after the overwrite. Gate behind explicit invocation (destructive to stage).
- **Risk if left:** bugs that only manifest on real-shaped data (volume, edge-case content, multi-household) slip through stage UAT and surface in prod. If built carelessly, the inverse risk dominates: a mis-pointed refresh that writes *into prod*, or leaking unscrubbed client PII into a lower-trust environment.

## - **E2E smoke tests against stage** — no automated post-deploy check confirms stage is actually up and the golden flows work after a promotion.
- **Where:** stage Railway environment (the deployed SPA + API). Existing E2E lives in `Application/Frigorino.IntegrationTests/` (Reqnroll + Playwright + Postgres Testcontainers) but it drives a *locally-built* harness, not a deployed environment — there's no suite that hits the live stage URL. Ties into [[project_branch_workflow]] (stage→main FF promotion; stage hosts real client UAT).
- **Why deferred:** the Testcontainers E2E covers behavior pre-merge; a deployed-environment smoke check is a separate concern (deploy health, env wiring, real auth) that wants its own change rather than folding into a feature.
- **Plan:** a small, fast Playwright suite — a handful of golden flows (auth/login, household switch, create+view a list, add an item, view a recipe) — run against the live stage URL after each promotion. Decide: (1) **trigger** — GitHub Actions post-deploy step, a Railway deploy hook, or manual/scheduled; (2) **auth** — real Firebase login needs a dedicated stage test account + credentials in CI secrets (DevAuth bypass is local-only, not on deployed stage); (3) **data hygiene** — smoke tests mutate real stage data, so either assert-only/read-mostly flows or create-and-clean-up with clearly-marked test entities (don't pollute the client's UAT view). Keep it a *smoke* suite (seconds, not the full IT matrix) so it gates promotion without slowing it. Note: assert on testids/`data-*`, never translated text ([[feedback_test_assertions_no_translated_text]]); use retrying `Expect(...).ToBeVisibleAsync()` ([[feedback_playwright_assertions]]).
- **Risk if left:** a broken deploy (env var missing, migration failed, auth misconfigured, SPA not served) is discovered by the real client on stage instead of by an automated gate — exactly the kind of env-wiring regression unit/IT tests can't catch (see [[feedback_verify_with_full_tests_and_docker]]).

## - **Railway Postgres 16 → 18 upgrade** — schedule a side-by-side dump/restore; no in-place path exists on Railway.
- **Where:** Railway Postgres service (managed; no repo change needed). Connection wiring: `ConnectionStrings__Database` env var on the `Frigorino.Web` Railway service, currently `${{Postgres.DATABASE_URL}}`. Researched 2026-05-23.
- **Why deferred:** PG16 is supported through Nov 2028; no immediate security/perf driver. Requires brief write downtime + a rollback window with the old service paused — wants explicit scheduling, not folded into an unrelated change. Stage cutover first per [[project_branch_workflow]].
- **Plan:** Side-by-side migration. (1) Add new service from Railway's [Deploy PostgreSQL 18](https://railway.com/deploy/postgresql-18-1) template; confirm volume size ≥ PG16's. (2) Scale `Frigorino.Web` to 0 replicas to quiesce API writes. (3) Run one-shot [railway-postgres-migration](https://railway.com/deploy/railway-postgres-migration) with `SOURCE_DATABASE_URL=${{Postgres.DATABASE_URL}}` + `TARGET_DATABASE_URL=${{Postgres18.DATABASE_URL}}` — streams `pg_dump --no-owner --no-privileges | psql`, no intermediate files (fine at current scale; alternative [postgres-migrator](https://railway.com/deploy/postgres-migrator) supports parallel `pg_restore` + row-count validation but its `PG_VERSION` build arg ships 15/16/17 client tools only — fork needed for v18). (4) Verify on PG18: row counts on `Users`/`Households`/`Lists`/`ListItems`/`Inventories`/`InventoryItems`/`UserHouseholds`/`__EFMigrationsHistory` and `SELECT schemaname, count(*) FROM pg_tables WHERE schemaname IN ('public','hangfire') GROUP BY 1`. (5) Repoint `ConnectionStrings__Database` to `${{Postgres18.DATABASE_URL}}`; Railway auto-redeploys; `MigrateAsync()` runs as no-op since `__EFMigrationsHistory` came over. (6) Smoke test golden flows. (7) Keep PG16 paused ≥24h for rollback (flip env back), then delete. PG18 breaking changes audited against codebase — none apply (no triggers / partitions / FTS / pg_trgm / raw COPY / MD5 auth; Npgsql 8 speaks SCRAM). Reference docs for the next session: [Railway PG overview](https://docs.railway.com/databases/postgresql), [PG18 release notes E.4](https://www.postgresql.org/docs/18/release-18.html), [PG18 pg_upgrade](https://www.postgresql.org/docs/current/pgupgrade.html), [Upgrading a Cluster 18.6](https://www.postgresql.org/docs/current/upgrading.html), [Station Q&A confirming no in-place path](https://station.railway.com/questions/upgrade-railway-postgres-v15-to-v16-5200d6dc), [PG18 release announcement](https://www.postgresql.org/about/news/postgresql-18-released-3142/).
- **Risk if left:** missing PG18 wins (native `uuidv7()`, async I/O, statistics-preserving `pg_upgrade` that would smooth the *next* bump). Drift accumulates — when PG16 EOL nears or PG19 ships, the jump grows and more breaking changes need re-auditing.

## - **List-level extraction poll instead of single-item poll** — removes the temp-id reconciliation and the "only the last add polls" gap in one move.
- **Where:** `Application/Frigorino.Web/ClientApp/src/features/lists/items/useExtractionPoll.ts`, `useCreateListItem.ts` (the `tempId = Date.now()` → real-id swap in `onSuccess`), `features/lists/pages/ListViewPage.tsx` (`pendingExtraction` is a single slot; rapid successive adds overwrite it so earlier rows show stale raw text until a debounced refetch). Surfaced in the 6-hat review (Black + Green hats).
- **Why deferred:** the single-item poll + optimistic temp-id reconciliation works correctly for the common one-at-a-time add; the multi-add gap is acknowledged "v1" behavior, not a regression.
- **Plan:** drop the per-item `getItem` poll; instead `refetchInterval` the existing `getItems` list query for a bounded window after any digit-bearing add, comparing each row's `quantity` to detect arrival. The indicator is already rendered per-row from list data. This makes `pendingExtraction` a set (or just "poll the list while any add is in flight"), and deletes the `tempId` swap entirely. Trade-off: refetches the whole (small) list vs one item.
- **Risk if left:** after rapid multi-item adds, earlier extracted quantities don't reflect until the ~1s debounced invalidation (which keeps resetting if the user keeps typing); the temp-id `Date.now()` is also a latent duplicate-key risk on same-millisecond adds.

## - **Household context is implicit (LastActiveHouseholdId) instead of carried per request** — most APIs infer the target household from session/last-active state rather than an explicit parameter.
- **Where:** `Application/Frigorino.Infrastructure` household-context resolution (`ICurrentHouseholdService` — active id in HTTP session, persisted to `User.LastActiveHouseholdId` as durable fallback) and every household-scoped slice in `Application/Frigorino.Features` that reads the active household instead of taking a household id. See "Multi-tenant household context" in `CLAUDE.md`.
- **Why deferred:** the implicit active-household model works for the single-window, one-household-at-a-time case it was designed for; making household explicit touches the URL/route shape and many slices, so it wants its own deliberate change rather than being folded into a feature.
- **Why it matters:** any flow that targets a household *other* than the active one breaks — most concretely the cross-household notification deep-link bug in `BUGS.md` ("Inventory fails to load when opening a notification for a non-active household"), where the link points at Household B but the request still resolves against active Household A. It also blocks multi-tab/multi-household concurrent use.
- **Plan:** make household context explicit rather than ambient. Options to weigh: (1) household id in the route path (`/households/{householdId}/inventories/...`) so every scoped request is self-describing; (2) household id as a required header/query param resolved by `ICurrentHouseholdService` per request, with last-active only as a default when omitted; (3) keep session active-household but have deep-links switch the active household before issuing the scoped query. Whichever path, the deep-link should carry the target household id (fixes the linked bug).
- **Risk if left:** cross-household actions (notifications, shared links, future multi-tab) silently hit the wrong household or fail to load; the bug above recurs in any new feature that links to a non-active household.

## - **Adopt implicit save-on-change across the app** — make "edit surfaces persist as you go" the default, not just recipes.
- **Where:** form/edit surfaces that still use an explicit Save (and sometimes Cancel) button — e.g. household create/edit, member management, list/inventory metadata edit, and any future edit screens. The recipe edit page (`Application/Frigorino.Web/ClientApp/src/features/recipes/pages/RecipeEditPage.tsx` + `components/EditRecipeForm.tsx`) is the first surface to move to implicit save (see `docs/superpowers/specs/2026-06-14-recipe-view-edit-refinement-design.md`); item lists (lists/inventories/recipes) already save immediately via optimistic mutations.
- **Why deferred:** the recipe refinement only converts the recipe edit view; rolling implicit save out to every form is a broader UX + consistency pass that wants its own change rather than being smuggled into one feature.
- **Plan:** generalize the pattern proven on the recipe edit page — seed fields once on mount (keyed by entity id), debounce changes (~600 ms) + flush on blur + clear pending timer on unmount, gate the save on per-field validation (don't PUT invalid data; show inline errors and keep the last valid value), and surface a light saving/saved status instead of a Save button. Extract a reusable `useDebouncedAutoSave` (or `useDebouncedCallback`) hook once a second surface needs it (no generic value-debounce hook exists today — only `useDebouncedInvalidation`, which is invalidation-specific). Audit each converted form for the "Cancel = discard" affordance users may expect to lose, and decide per surface whether back-without-undo is acceptable.
- **Risk if left:** inconsistent mental model — some screens save on type, others need an explicit Save, so users lose edits they assumed were saved (or hunt for a Save button that isn't there). The longer mixed conventions persist, the more each new form re-litigates the choice.

## - **`npm audit`: 18 moderate findings with no clean forward fix** — two upstream chains, accepted as low-risk pending upstream releases. Audited 2026-06-21.
- **Where:** `Application/Frigorino.Web/ClientApp/package-lock.json`. Two independent chains: (1) **runtime** — `@opentelemetry/core` <2.8.0 ([GHSA-8988-4f7v-96qf](https://github.com/advisories/GHSA-8988-4f7v-96qf), unbounded memory in W3C Baggage propagation) inherited up the OTel tree into our direct `@grafana/faro-web-sdk` + `@grafana/faro-web-tracing`; (2) **build-time only** — `js-yaml` ≤4.1.1 ([GHSA-h67p-54hq-rp68](https://github.com/advisories/GHSA-h67p-54hq-rp68), quadratic-complexity DoS) reached solely via the `@hey-api/openapi-ts` devDependency → `@hey-api/json-schema-ref-parser`.
- **Why deferred:** no fix that improves security is safely available. (1) Patched OTel core 2.8.0 exists, but we're already on the newest Faro (2.7.1) and it still pins core <2.8.0, so `npm audit` reports no fix for the Faro packages — blocked on a Grafana release built against OTel core ≥2.8.0. Practical risk low: the DoS is in baggage the RUM SDK *generates*, not attacker input it parses. (2) We're on the latest hey-api (0.98.2); npm's only "fix" is `--force` → **downgrade to 0.97.0** (breaking, backwards). hey-api is a build-time codegen tool parsing our *own* `openapi.json` — zero runtime exposure. `npm audit fix` (non-force) is a confirmed no-op; **never run `--force` here.**
- **Plan:** re-run `npm audit` each deps-bump cycle. Clear chain (1) when a Faro release depends on `@opentelemetry/core` ≥2.8.0 (then a normal `@grafana/faro-*` minor bump resolves it). Clear chain (2) when a `@hey-api/openapi-ts` release bumps `@hey-api/json-schema-ref-parser`/`js-yaml` forward past the vulnerable range. A riskier interim option for (1) — pin OTel core 2.8.0 (+ the lockstep OTel set) via npm `overrides` — was deliberately skipped: it runs Faro 2.7.1 against an undeclared OTel minor and needs a runtime-init verify for a low-exploitability moderate.
- **Risk if left:** all 18 are moderate, 0 high/critical. Runtime exposure is a low-exploitability DoS in self-generated telemetry baggage; the js-yaml finding has no runtime footprint at all (build-time, own input). Drift accumulates if upstreams ship fixes and we don't re-check.
