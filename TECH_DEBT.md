## - **Railway Postgres 16 → 18 upgrade** — schedule a side-by-side dump/restore; no in-place path exists on Railway.
- **Where:** Railway Postgres service (managed; no repo change needed). Connection wiring: `ConnectionStrings__Database` env var on the `Frigorino.Web` Railway service, currently `${{Postgres.DATABASE_URL}}`. Researched 2026-05-23.
- **Why deferred:** PG16 is supported through Nov 2028; no immediate security/perf driver. Requires brief write downtime + a rollback window with the old service paused — wants explicit scheduling, not folded into an unrelated change. Stage cutover first per [[project_branch_workflow]].
- **Plan:** Side-by-side migration. (1) Add new service from Railway's [Deploy PostgreSQL 18](https://railway.com/deploy/postgresql-18-1) template; confirm volume size ≥ PG16's. (2) Scale `Frigorino.Web` to 0 replicas to quiesce API writes. (3) Run one-shot [railway-postgres-migration](https://railway.com/deploy/railway-postgres-migration) with `SOURCE_DATABASE_URL=${{Postgres.DATABASE_URL}}` + `TARGET_DATABASE_URL=${{Postgres18.DATABASE_URL}}` — streams `pg_dump --no-owner --no-privileges | psql`, no intermediate files (fine at current scale; alternative [postgres-migrator](https://railway.com/deploy/postgres-migrator) supports parallel `pg_restore` + row-count validation but its `PG_VERSION` build arg ships 15/16/17 client tools only — fork needed for v18). (4) Verify on PG18: row counts on `Users`/`Households`/`Lists`/`ListItems`/`Inventories`/`InventoryItems`/`UserHouseholds`/`__EFMigrationsHistory` and `SELECT schemaname, count(*) FROM pg_tables WHERE schemaname IN ('public','hangfire') GROUP BY 1`. (5) Repoint `ConnectionStrings__Database` to `${{Postgres18.DATABASE_URL}}`; Railway auto-redeploys; `MigrateAsync()` runs as no-op since `__EFMigrationsHistory` came over. (6) Smoke test golden flows. (7) Keep PG16 paused ≥24h for rollback (flip env back), then delete. PG18 breaking changes audited against codebase — none apply (no triggers / partitions / FTS / pg_trgm / raw COPY / MD5 auth; Npgsql 8 speaks SCRAM). Reference docs for the next session: [Railway PG overview](https://docs.railway.com/databases/postgresql), [PG18 release notes E.4](https://www.postgresql.org/docs/18/release-18.html), [PG18 pg_upgrade](https://www.postgresql.org/docs/current/pgupgrade.html), [Upgrading a Cluster 18.6](https://www.postgresql.org/docs/current/upgrading.html), [Station Q&A confirming no in-place path](https://station.railway.com/questions/upgrade-railway-postgres-v15-to-v16-5200d6dc), [PG18 release announcement](https://www.postgresql.org/about/news/postgresql-18-released-3142/).
- **Risk if left:** missing PG18 wins (native `uuidv7()`, async I/O, statistics-preserving `pg_upgrade` that would smooth the *next* bump). Drift accumulates — when PG16 EOL nears or PG19 ships, the jump grows and more breaking changes need re-auditing.

## - **List-level extraction poll instead of single-item poll** — removes the temp-id reconciliation and the "only the last add polls" gap in one move.
- **Where:** `Application/Frigorino.Web/ClientApp/src/features/lists/items/useExtractionPoll.ts`, `useCreateListItem.ts` (the `tempId = Date.now()` → real-id swap in `onSuccess`), `features/lists/pages/ListViewPage.tsx` (`pendingExtraction` is a single slot; rapid successive adds overwrite it so earlier rows show stale raw text until a debounced refetch). Surfaced in the 6-hat review (Black + Green hats).
- **Why deferred:** the single-item poll + optimistic temp-id reconciliation works correctly for the common one-at-a-time add; the multi-add gap is acknowledged "v1" behavior, not a regression.
- **Plan:** drop the per-item `getItem` poll; instead `refetchInterval` the existing `getItems` list query for a bounded window after any digit-bearing add, comparing each row's `quantity` to detect arrival. The indicator is already rendered per-row from list data. This makes `pendingExtraction` a set (or just "poll the list while any add is in flight"), and deletes the `tempId` swap entirely. Trade-off: refetches the whole (small) list vs one item.
- **Risk if left:** after rapid multi-item adds, earlier extracted quantities don't reflect until the ~1s debounced invalidation (which keeps resetting if the user keeps typing); the temp-id `Date.now()` is also a latent duplicate-key risk on same-millisecond adds.

## - **Household context is implicit (LastActiveHouseholdId) instead of carried per request** — most APIs infer the target household from session/last-active state rather than an explicit parameter.
- **Where:** `Application/Frigorino.Infrastructure` household-context resolution (`ICurrentHouseholdService` — active id in HTTP session, persisted to `User.LastActiveHouseholdId` as durable fallback) and every household-scoped slice in `Application/Frigorino.Features` that reads the active household instead of taking a household id. See "Multi-tenant household context" in `CLAUDE.md`.
- **Why deferred:** the implicit active-household model works for the single-window, one-household-at-a-time case it was designed for; making household explicit touches the URL/route shape and many slices, so it wants its own deliberate change rather than being folded into a feature.
- **Why it matters:** any flow that targets a household *other* than the active one breaks — most concretely the cross-household notification deep-link bug in `BUGS.md` ("Inventory fails to load when opening a notification for a non-active household"), where the link points at Household B but the request still resolves against active Household A. It also blocks multi-tab/multi-household concurrent use.
- **Plan:** make household context explicit rather than ambient. Options to weigh: (1) household id in the route path (`/households/{householdId}/inventories/...`) so every scoped request is self-describing; (2) household id as a required header/query param resolved by `ICurrentHouseholdService` per request, with last-active only as a default when omitted; (3) keep session active-household but have deep-links switch the active household before issuing the scoped query. Whichever path, the deep-link should carry the target household id (fixes the linked bug).
- **Risk if left:** cross-household actions (notifications, shared links, future multi-tab) silently hit the wrong household or fail to load; the bug above recurs in any new feature that links to a non-active household.

## - **`UpdatedAt` overloaded as the checked-item retention clock** — any bulk `SaveChanges` silently resets the purge window, so checked items can't be aged out reliably.
- **Where:** `Application/Frigorino.Infrastructure/Tasks/DeleteInactiveItems.cs` (passes `li.UpdatedAt` as the "checked-off at" timestamp into `CheckedItemPurge.SelectExpiredItemIds`); `Application/Frigorino.Infrastructure/Tasks/CheckedItemPurge.cs` (compares `UpdatedAt < now - retention`); `ListItem.UpdatedAt` is the generic last-modified field auto-stamped on every `Modified` entity in `Application/Frigorino.Infrastructure/EntityFramework/ApplicationDbContext.cs:143-146`.
- **Why deferred:** the retention feature works for the common case (check an item, leave it, it ages out at 30 days); the overloaded-timestamp weakness only surfaced while investigating prod data, and a dedicated column is a schema change wanting its own deliberate migration + backfill rather than being folded into a bugfix.
- **Why it matters:** `UpdatedAt` means "last touched for any reason," not "last checked off," so anything that bumps it on a checked item resets that item's 30-day purge window. This already happened in prod: the now-removed `RecalculateSortOrderTask` (deleted in `547dc582`, 2026-05-16) swept every active list calling `RecalculateFullSortOrder` → one bulk `SaveChanges` per list → stamped `UpdatedAt = 2026-05-16` on *every* item in the list (visible in prod as whole-list-identical timestamps to the second, distinct per list). Every checked item that lived through that sweep had its retention clock reset regardless of when it was actually checked — which is why old checked items (created Sept 2025) still won't purge until ~mid-June 2026. The live `ExtractQuantityJob` (`Frigorino.Infrastructure/Services/ExtractQuantityJob.cs:73,79`) and any user reorder/edit bump `UpdatedAt` the same way. No current maintenance task does (`DeleteInactiveItems` uses `ExecuteDelete`, `BackfillProductClassification` writes only `Product`), but the design stays vulnerable to the next bulk `SaveChanges` over `ListItems`.
- **Plan:** introduce a dedicated nullable `CheckedAt` (or `StatusChangedAt`) `DateTime?` on `ListItem`, stamped **only** by the `List` aggregate when an item toggles to checked (`ToggleItemStatus`, plus the status-change branch of `UpdateItem`) and cleared when it goes back to unchecked. Repoint `DeleteInactiveItems`/`CheckedItemPurge` at `CheckedAt` instead of `UpdatedAt`. Migration: add the column nullable; backfill `CheckedAt = UpdatedAt` for currently-checked rows (`Status = true`) as a best-effort seed (accepts the May-16 contamination for existing rows — only delays their purge, never deletes early), `NULL` for unchecked. Keep a `null`/`default(DateTime)` guard in the pure selector so an unknown stamp never reads as "ancient → purge." The selector stays DB-free so it remains unit-testable without the InMemory provider.
- **Risk if left:** retention is effectively unbounded for any checked item touched by a sweep, reorder, quantity extraction, or future bulk migration — checked items accumulate instead of being purged at 30 days, and purge timing is non-deterministic w.r.t. real check-off time. The next bulk `SaveChanges` over `ListItems` silently resets every household's retention window again.

## - **Sparse sort-order scheme has no rebalancer left** — midpoint reorders can exhaust the gap and collide, with no recovery path.
- **Where:** `Application/Frigorino.Domain/Entities/List.cs:491` (`ReorderItem` midpoint insert: `afterItem.SortOrder + (beforeItem.SortOrder - afterItem.SortOrder) / 2`), `SortOrderCalculator.cs:9-13` (`UncheckedMinRange = 1_000_000`, `CheckedMinRange = 10_000_000`, `DefaultGap = 10_000`), `Application/Frigorino.Features/Lists/Items/GetItems.cs:45-46` (orders by `Status, SortOrder` with no tertiary tiebreaker). Orphaned rebalancer that's still in the tree but has no caller: `List.CompactItems()` + `POST /compact` (`Frigorino.Features/Lists/Items/CompactItems.cs`) + the unused `useCompactListItems` hook.
- **Why deferred:** the sparse/fractional-index scheme keeps reorders O(1) (one item moves, nobody else is renumbered), and real lists rarely reorder into the same tight spot enough to exhaust a 10k gap, so it hasn't bitten in practice. The rebalancer that backed it was removed for unrelated reasons (manual "Compact" button removed Aug 2025; `RecalculateSortOrderTask` startup sweep removed `547dc582` 2026-05-16 — see the `UpdatedAt`-retention item above for why that sweep was bad).
- **Why it matters:** midpoint insertion halves the available gap each time you drop an item into the same slot (`10000 → 5000 → … → 1 → 0`), so ~13 reorders into one region collapse the gap to 1 where `(1)/2 == 0` and the moved item lands on the **exact same SortOrder** as its neighbor. With no tiebreaker in `GetItems`, equal SortOrders sort nondeterministically → the reorder "won't stick" / the list reshuffles on refetch, and there is now **no path (auto or manual) to re-space the list** so the degradation is permanent for that region. Secondary, long-horizon: new unchecked items append `last + 10k` (walking up from 1M) and checked items prepend `first - 10k` (walking down from 10M); the deleted compaction was what pulled both back into their base ranges, so heavily-churned lists slowly drift toward range overlap.
- **Decision driver — planned real-time multi-user sync changes the criterion** from *write cost* to *conflict resolution*: when two household members reorder the same list at once, must both moves survive, or is last-write-wins acceptable? The bounded-integer scheme is the *weak* form of fractional indexing — for <1000-item household lists it was overkill on write-cost grounds, but once concurrent reordering is real, an order-array-on-parent (whole list = one conflict unit, last-write-wins clobbers a concurrent move) and integer renumber (interleaved renumbers corrupt the total order) are both poor fits, while proper string fractional indexing becomes the *right* tool — not for write cost, but because independent per-item keys merge concurrent reorders cleanly.
- **Plan — two tiers:**
  - **(Near-term band-aid, keep current scheme)** re-arm rebalancing **on-demand, not as a periodic sweep**. In `ReorderItem`, detect gap exhaustion (`beforeItem.SortOrder - afterItem.SortOrder <= 1`) and recompact **just that one list, in the same request** (reuse `CompactItems`/`GenerateCompactedSortOrders`, then place the moved item). Self-heals only when needed, never a startup batch. Add a `.ThenBy(i => i.Id)` tiebreaker to `GetItems` (and the inventory equivalent) against residual ties. Re-stamps `UpdatedAt` on renumbered items — only stops mattering once retention moves to the dedicated `CheckedAt` field (item above), so do these together.
  - **(Recommended refactor — candidate C, string fractional indexing)** replace the integer `SortOrder` with a lexicographic string rank key (LexoRank-style; e.g. the `fractional-indexing` package or a small in-house impl). Keys like `"a0" < "a0V" < "a1"` have unbounded precision (never exhaust, no rebalancer ever) and one-row writes per reorder. This is the canonical primitive for collaborative sequences (Figma/Linear/Jira/Trello; Yjs uses a variant) and the only option that satisfies *both* constraints we care about — no N-row renumber **and** no whole-list conflict unit — making it the best fit for the real-time roadmap. **One gotcha to get right:** two users inserting at the same slot can compute the same key → collision; break ties with a stable per-actor suffix/salt (don't ship the tie-free-but-naive version, which is the same mistake as the current integer midpoint). Migration: add a `Rank` string column, backfill from current `Status`+`SortOrder` order via `GenerateKeysBetween`, repoint `GetItems` ordering + `ReorderItem` to the key, then drop `SortOrder`/`SortOrderCalculator`/`CompactItems`/`/compact`. Two status sections map to two key spaces (or one space + filter by `Status`).
- **Pick by:** if reorders stay rare and last-write-wins is acceptable, the band-aid (or order-array) is enough; if shared lists will see real concurrent reordering, go straight to C and skip iterating the integer scheme.
- **Risk if left:** a user who fine-tunes ordering in one spot of a list eventually hits silent SortOrder collisions, after which drag-reorder appears broken (items jump back / random order after refresh) with no way to fix it from the UI; over a very long horizon the two status ranges can converge. Once real-time sync lands on top of the current scheme, concurrent reorders will also clobber each other / corrupt order, with no clean merge path — addressing ordering before (or with) sync avoids reworking it twice.
